第一章：计算机网络概述

什么是计算机网络

计算机网络是通信技术与计算机技术紧密结合的产物。

通信系统模型：

  • 信源
  • 发送设备
  • 信道
    • 噪声源
  • 接收设备
  • 信宿

计算机网络本质上就是一种通信网络，是一种互联的，自治的计算机集合。

计算机网络协议

网络协议是为了进行网络中数据交换所建立的规则标准和约定

协议的三要素：

  • 语法
  • 语义
  • 时序

计算机网络的结构

  • 网络边缘
    • 主机
    • 网络应用
      • 客户端/服务器模型
      • 对等应用模型
  • 接入网络，物理介质
    • 通信链路
  • 核心网络
    • 交换单元(交换机/路由器等)
  • 协议

接入网络

  • 数字用户线路（DSL）

    • 多路复用
    • 利用已有的电话线连接
    • <2.5Mbps上行
    • <24Mbps下行
    • FDM
      • 50kHz-1MHz用于下行
      • 4-50kHZ用于上行
      • <4kHz用于电话

  • 电缆网络（HFC,混合光纤同轴电缆）

    • 频分多路复用
      • 在不同频带（载波）上传输不同频道
    • 利用已有的电视线缆连接

交换机和路由器

  • 负责转发
  • 根据存储的路由表进行转发

Internet结构

  • 一级ISP连接各大网段
  • 一级ISP之间通过对等链路互联
  • 一级ISP下还有区域ISP
  • 内容提供商（CPN）可就近构建自己的网络

数据交换

  • 为什么需要链路交换？

因为N^2链路问题，若N个主机全部需要互相链接，那么需要的链路数为N*(N-1)/2，当主机数较大时所需链路数也极大。使用交换设备可以使每个主机只连接交换设备即可实现数据的通信。当单个交换设备不足以处理大量主机时，可以将多个交换设备组成交换网络。

交换的分类

  • 电路交换
    • 电话网络
    • 步骤：建立连接、通信、释放连接
    • 独占资源
  • 报文交换
    • 发送源直接向目标发送信号整体
  • 分组交换
    • 报文分拆出一系列较小的数据包
    • 需要报文的分拆和重组
    • 报文交换和分组交换都采用存储转发交换方式
    • 适用于突发数据传输网络
    • 可能产生拥塞

多路复用技术

  • 频分复用(FDM)
  • 时分复用
  • 波分复用
    • 就是光的频分复用
  • 码分复用(CDM)

分组交换的报文交付时间

  • 报文：M bits
  • 链路带宽：R bps
  • 分组长度：L bits
  • 跳步数：h
  • 路由器数：n
  • 交付时间为：T=M/R+(nL)/R

计算机网络性能

速率

数据传输速率，也称为比特率

单位有bps，kbps，mbps等

通常指额定速率或标定速率

带宽

计算机网络中指数据信道能传输的最大数据率

单位和速率一致

时延/延迟

缓存中存储正在发送的分组和排队的分组，若缓存被待传输的分组占满则会产生丢包现象

四种分组延迟：

  • 结点处理延迟
    • 差错检测
    • 确定输出链路
  • 排队延迟
    • 链路带宽R
    • 分组长度L
    • 平均分组到达速率
    • 流量强度La/R
      • 接近于0：平均排队延迟很小
      • 接近于1：平均排队延迟很大
      • 大于1：超出传输能力，延迟无限大
  • 传输延迟
    • 分组长度L
    • 链路带宽R
    • L/R
  • 传播延迟
    • 物理链路长度d
    • 信号传播速度s
    • d/s

几种衡量指标

  • 时延带宽积
    • 传播时延（dprop） x 带宽（R）
  • 丢包率
    • 丢包数/已发送分组数
  • 吞吐量
    • 即时吞吐量：给定时刻的速率
    • 平均吞吐量：一段时间的平均速率

第二章：网络应用

网络应用体系结构

客户机/服务器

服务器

  • 24小时提供服务
  • 永久访问地址和域名
  • 利用大量服务器实现可扩展性

客户机

  • 与服务器通信
  • 间歇性接入网络
  • 可使用动态IP地址
  • 不会与其他客户机直接通信

P2P

  • 没有永远在线的服务器
  • 任意端系统/节点可直接通信
  • 节点间歇接入网络
  • 节点可能改变IP地址
  • 高度可伸缩
  • 难于管理

混合结构

  • 两种连接方式并存
  • 例如Napster

网络应用的进程通信

进程间通信

  • 依赖Socket进行通信
  • 传输基础设施向进程提供API

如何寻址进程

  • 不同主机的进程之间通信必须有标识符
  • 查找特定主机需要IP地址
  • 查找特定进程需要端口号
  • 进程的标识符为IP地址+端口号

应用层协议

  • 公开协议
    • 由RFC制定
    • 允许互操作
  • 私有协议
    • P2P共享应用

应用层协议的内容

  • 类型
  • 语法格式
  • 字段的语义
  • 规则

网络应用的服务需求

  • 可靠性
  • 带宽
  • 时延

Internet传输层服务模型

TCP

  • 面向连接
  • 可靠的传输
  • 流量控制
  • 拥塞控制
  • 不提供时间/延迟保障
  • 不提供最小带宽保证

UDP

  • 无连接
  • 不可靠的数据传输
  • 不提供TCP所具有的性质

特定网络应用及协议

  • URL/统一资源定位器
    • protocol://domain(ip_address):port/path

HTTP

  • 超文本传输协议
  • 客户端/服务器模式
  • TCP传输模式
  • 无状态协议
    • 不维护任何有关客户端过去所发的请求
  • HTTP连接的两种类型
    • 非持久
      • 每次传输只允许一个对象
      • HTTP1.0
      • 需要为每个TCP连接开销资源
      • 每个对象需要2个RTT
    • 持久
      • 允许传输多个对象
      • HTTP1.1开始使用持久连接
      • 发送响应后保持TCP连接的打开状态
      • 后续的HTTPS请求可通过此连接进行传输
      • 每个需要1个RTT
      • Streamng Mode：遇到引用对象就立即发出请求
        • 此模式下收到所有引用对象只需花费1个RTT
  • 响应时间
    • RTT（Round Trip Time）：从客户端发送一个很小的数据包并返回所经历的时间
    • 2RTT+传输时间
  • 消息格式
    • 请求消息
      • Request Lines(请求行)：GET/POST/HEAD+URL+HTTP版本
      • Header Lines(头部行)：可扩展，包括UA/Host/Connection和Refer等
      • Carriage Return：表示请求内容结束
      • Entity Body：需要传输的数据
    • 上传消息
      • POST：大多数通过<form>表格发出带有目标数据的POST数据包
      • URL：通过将参数包含在Request Lines的URL字段发出
    • 响应消息
      • Status Lines（状态行）：HTTP版本+状态码+状态号
      • Header Lines（头部行）：包含Connection/Date/Server等
      • Carriage Return：表示请求内容结束
      • Entity Body：需要传输的数据
  • 方法类型
    • HTTP1.0
      • GET
      • POST
      • HEAD
        • 让服务器只发送请求头
    • HTTP1.1
      • GET/POST/HEAD
      • PUT
        • 将Entity Body中的文件上传到URL指定的位置
      • DELETE
        • 删除URL指定的文件
  • Cookies技术
    • 网站为了辨别用户身份进行session trace而储存在用户本地的数据（通常被加密）
    • cookie组件
      • HTTP响应消息的cookie头部行
      • HTTP请求消息的cookie头部行
      • 保存在客户机上的cookie文件，由浏览器管理
      • web服务器的后台数据库
    • 用途
      • 身份认证
      • 购物车
      • 算法推荐
      • email
  • WEB缓存与代理服务器
    • 在不直接访问目标服务器的前提下满足客户端的HTTP请求
    • 实现方法
      • 用户向代理服务器发送GET请求
      • 代理服务器检查是否有对应的缓存
        • 若有则对请求予以响应
        • 若没有则向目标服务器发送GET并转发响应给客户端
    • WEB缓存性能分析
      • LAN利用率：公网通讯对象平均大小*请求数/LAN的每秒数据传输量
      • 链路利用率：公网通讯对象平均大小*请求数/WAN的每秒数据传输量
      • 平均总延迟：互联网延迟+访问延迟+局域网延迟
      • 使用WEB缓存服务器可有效缓解低WAN链路速率造成的传输缓慢问题
    • 解决WEB缓存服务器与服务器同步问题
      • 条件性GET方法
        • 如果缓存有最新版本，则不需要发送请求对象
      • 在HTTP请求消息中声明所持有版本的日期
        • if-modified-since:<date>
      • 如果缓存版本是最新的，则服务器不返回对象
        • HTTP/1.0 304 Not Modified

SMTP/POP/IMAP

  • 组成结构
    • 邮件客户端
    • 邮件服务器
    • SMTP协议
  • 使用TCP进行email的可靠传输
  • 端口25
  • 传输过程
    • 握手
    • 消息传输
      • Header
        • To
        • From
        • Subject
      • Body
        • 只能包含7位ascii码
    • 关闭
  • 传输步骤
    • HELO <client_address>
    • MAIL FROM: <sender address>
    • RCPT TO: <receiver address>
    • DATA
    • <data>
    • .
    • QUIT
  • 使用CRLF.CRLF作为传输终止标志
    • CRLF为换行符
  • 多媒体扩展问题
    • 在邮件头部添加额外的MIME参数用于识别类型
    • 添加的行
      • MIME-Version
      • Content-Transfer-Encoding
      • Content-Type
  • 邮件访问协议
    • POP
      • Post Office Protocol
      • 认证/授权/下载
      • 下载并删除和下载并保持模式
      • 无状态
    • IMAP
      • Internet Mail Access Protocol
      • 更多功能
      • 更复杂
      • 能够操作服务器的数据
    • HTTP

DNS

  • 多层命名服务器构成的分布式数据库
  • 应用层上完成对名字的解析
  • 为web服务器提供负载均衡
  • 域名向IP地址的翻译
  • 主机别名
  • 查询与回复型通信
  • 消息格式相同
  • DNS解析次序
    • 本地域名解析（hosts文件）
    • 若没有，转向根域名服务器
      • DNS劫持通常会伪造成根域名服务器来劫持请求并返回错误的映射，最常见于某米某讯原厂固件wifi自带的广告
    • 若还是没有，转向权威域名服务器
      • 中国国家防火墙（GFW）DNS污染以及运营商DNS劫持的重点攻击对象
      • 通常会劫持返回的映射到一个无法访问的IP地址从而造成用户connection timedout
      • 若用户开启了DoH（DNS over HTTPS）或者DoT（DNS over TLS）则GFW会使用TCP重置攻击，通过恶意拦截并向TCP报文中注入REST标志来造成用户connection reset
    • 获得映射
    • 返回映射
  • 顶级域名服务器
    • 负责org/com/net/edu等顶级域名和国家域名
  • 本地域名解析服务器
    • 每个网络供应商（ISP）都有
  • 迭代查询
    • 本地DNS服务器发送多次请求
    • 踢皮球式查询法
  • 递归查询
    • 主机和本地DNS都只发送一次请求，其余的都由后续服务器处理
    • 传火式查询法
  • DNS缓存
    • 只要DNS解析服务器获得了IP/域名映射就会缓存此映射
    • 一段时间后失效并删除，随后缓存新的映射
  • DNS记录
    • 资源记录（Resource Record）
      • Type A
        • Name: Host Name
        • Value: IP Address
      • Type NS
        • Name: Domain
        • Value: Authority DNS server’s host name
      • Type CNAME
        • Name: Alternative name of a domain
        • Value: Real name of a domain
      • Type MX
        • Value是与name对应的邮件服务器
  • 消息结构
    • 消息头部
      • 16位查询编号
    • 查询
    • 回复
    • 权威回答
    • 补充信息

P2P

  • 特点
    • 没有服务器
    • 任意端系统直接通信
    • 结点阶段性接入Internet
    • 结点可能更换IP地址
  • 数据传输速率比客户端/服务器模式快
    • 客户端/服务器：dcs = max{NF/us, F/min(di)}
      • F：文件大小
      • N：副本数
      • us：上传速度
      • di：下载速度
    • P2P：dcs = max{F/us, F/min(di)i, NF/(us+εui)}
      • us+εui最快可能上传速率
  • 应用
    • BitTorrent
    • IPFS
    • 区块链
    • 分布式计算
  • 结构
    • peer
      • 上传与下载者
      • “做种（Seeding）”就是代表某个peer向整个p2p网络分享文件的行为
      • 从其他peer下载chunk的同时也会向其他peer共享本地以下载但对方没有的chunk
    • Tracker
      • 跟踪参与Torrent的结点
  • P2P的索引
    • 信息到结点位置的映射(IP+端口号)
    • 文件共享
      • 利用索引动态跟踪结点共享的文件位置
      • 结点需要告诉索引它拥有哪些文件
      • 结点搜索索引从而知道能够获得哪些文件
    • 即时通讯
      • 负责将用户名映射到位置
      • 开启IM时，需要通知索引它的位置
      • 结点搜索索引，确定用户的IP地址
    • 集中式索引
      • 结点需要向中央服务器提供自己的IP地址和文件列表
      • 单点失效问题
      • 性能瓶颈
      • 版权问题
    • 泛洪式索引
      • 完全分布式架构
      • 每个结点都只对它共享的文件进行索引
      • 通过已有的TCP连接发送
      • 结点转发查询消息
      • 如果查询命中，则利用反向路径发回查询结点
    • 层次型覆盖网络
      • 混合型
      • 分为超级结点和普通结点
      • 超级结点和结点维持TCP连接
      • 私有应用层协议
      • 索引分布在超级结点上

IP

  • 提供寻址服务
  • 传输格式
    • 版本（4位）
      • IPv4/IPv6
    • 首部长度（4位）
    • 优先级与服务类型（4位）
      • 是否对数据包进行特殊处理
    • 总长度（16位）
      • 头部长度+可选项和数据长度
      • 空请求为20字节
    • 标识符（16位）
      • 用于数据包的分片与组合
    • 标志（13位）
      • 用于数据包的分片与组合
    • 段偏移量（13位）
      • 用于数据包的分片与组合
    • TTL（8位）
      • 原指生存时间
      • 实际代表数据包能够经过的最多路由器数量
    • 协议号（8位）

第三章：传输层

传输层协议

  • 为不同host上的进程提供了逻辑通信机制
  • 能够将从应用层接收到的消息分割成多个seg并传输

与网络层的异同

  • 提供应用进程之间的逻辑通信机制
    • 网络层提供主机之间的逻辑通信机制
  • 位于网络层之上
  • 依赖于网络层服务
  • 对网络层服务进行增强

Internet传输层协议

  • TCP
    • 参考前文
  • 多路复用
    • 如果某一层的一个协议对应直接上层的多个协议/实体，则需要复用/分用
    • 接收端多路复用
    • 发送端多路复用
  • 多路分用
    • 无连接的多路分用
      • 采用UDP传输方式
      • UDP的Socket用二元组标识
        • 目的IP地址和目的端口号
      • 主机收到UDP段后检查端口号并将它绑定到对应端口的Socket上
      • 来自不同源IP地址和/或源端口号的IP数据包被导向同一个Socket
    • 有连接的多路分用
      • 采用TCP连接方式
      • TCP的Socket采用四元组标识
        • 源IP地址
        • 源端口号
        • 目的IP地址
        • 目的端口号
      • 接收端利用所有的四个值将Segment导向合适的Socket
      • 服务器端可能同时支持多个人TCP Socket
        • 每个都有自己的四元组标识
      • 服务器为每个客户端开设不同的Socket
  • UDP
    • 参考前文
    • 基于Internet IP协议
    • UDP段传输不可靠，可能丢失或乱序
    • 无连接，不需要握手
    • UDP优势
      • 无需建立连接（延迟低）
      • 实现简单，无需维护
      • 头部开销少
      • 无拥塞控制
    • 主要应用
      • 流媒体
      • DNS
      • SNMP
    • 使UDP上实现可靠数据传输
      • 应用层上增加可靠性机制
      • 应用特定的错误恢复机制
    • UDP校验和
      • 检验UDP传输错误
